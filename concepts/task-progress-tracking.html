Task Progress Tracking

Definition

Task progress tracking is a real-time mechanism that provides granular visibility into long-running task execution, allowing frontends to monitor task status and intermediate results without frequent polling.

Purpose

The CE Orchestrator implements comprehensive task progress tracking to address limitations in the original system:

  1. Real-time Updates - WebSocket-based push instead of polling
  2. Granular Progress - Step-by-step progress with percentages
  3. Container Operation Tracking - Asynchronous container operation monitoring
  4. Intermediate Results - Streaming output and artifacts during execution

Architecture

Progress Events

Location: src/services/progress-emitter.ts

Progress events follow a structured format:

interface ProgressEvent {
  eventId: string;
  timestamp: string;
  taskId: string;
  projectId: string;
  type: 'task_queued' | 'task_started' | 'step_progress' | 'output' | 'artifact' | 'task_completed' | 'task_failed' | 'task_cancelled';
  data: {
    status?: string;
    progress?: {
      current: number;
      total: number;
      percentage: number;
      message: string;
    };
    output?: string;
    artifacts?: TaskArtifact[];
    error?: string;
    metrics?: {
      elapsed: number;
      estimatedRemaining?: number;
    };
  };
}

WebSocket Connection

Location: src/controllers/ProgressController.ts

Clients connect via WebSocket:

WS /api/progress/subscribe/{projectId}

Subscription Format

{
  "action": "subscribe",
  "taskIds": ["task-1", "task-2"],
  "types": ["task_started", "step_progress", "task_completed"]
}

API Changes

Container Operations

Synchronous operations remain for quick actions, long operations become asynchronous:

// Async container operation
POST /flexy/containers/{id}/start-async -> { operationId: string }

// Query operation status
GET /flexy/operations/{operationId} -> { status, progress }

Workflow Progress

interface WorkflowProgress {
  workflowId: string;
  status: 'running' | 'paused' | 'completed' | 'failed';
  currentPhase: number;
  phases: Array<{
    id: string;
    name: string;
    status: 'pending' | 'running' | 'completed' | 'failed';
    progress: number;
    message: string;
  }>;
}

Configuration

# WebSocket Configuration
PROGRESS_WS_ENABLED=true
PROGRESS_WS_MAX_CONNECTIONS=100
PROGRESS_WS_HEARTBEAT_INTERVAL=30000

# Progress Reporting
PROGRESS_REPORT_INTERVAL=1000    # milliseconds
PROGRESS_OUTPUT_CHUNK_SIZE=2048  # characters

Security Considerations

  1. Authentication - WebSocket connections require Bearer Token
  2. Authorization - Users can only subscribe to projects they have access to
  3. Connection Limits - Max concurrent connections per user
  4. Rate Limiting - Connection rate limiting

Performance Considerations

  1. Event Emitter Pattern - No polling overhead
  2. Batch Updates - Reduce network load by batching progress updates
  3. Auto-Reconnect - Client-side reconnection handling
  4. Compression - Compress large output data

Backward Compatibility

  • Existing polling-based API endpoints remain unchanged
  • WebSocket push is an optional feature
  • Synchronous container operations preserved
  • Async versions are additions, not replacements
